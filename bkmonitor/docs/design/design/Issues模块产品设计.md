## 1. 模块定位

### 1.1 解决的核心问题

当前告警系统的生命周期终点是指标层面的状态恢复（RECOVERED/CLOSED），但**监控感知到的恢复 ≠ 问题真正被解决**。

典型场景：服务内存溢出告警触发 → 重启后告警恢复 → 但导致 OOM 的代码 bug 仍未修复，系统无法感知，也无法追踪。

Issues 模块在告警生命周期结束后提供一个**人工驱动的跟踪层**：

```
告警触发 ──→ 告警恢复（监控层面结束）
                ↓
           Issue 创建 ──→ 人工跟进 ──→ 确认完成（人工层面结束）
```

Issue 的生命周期由人来驱动，不依赖指标状态。

### 1.2 与相关系统的关系

| 系统 | 定位 | 与 Issues 的关系 |
|------|------|----------------|
| **应急系统** | 管理流程补充，解决影响业务的故障升级处置 | 告警可以升级应急，也可以只触发 Issue；二者并行不互斥 |
| **ITSM / TAPD / GitHub Issues** | 具体研发/运维执行事项 | Issues 的下游；一个 Issue 可关联多个外部事项（1:N） |
| **告警中心** | 感知层，"发生了什么" | Issues 是告警中心的延伸，附着在告警/策略上的跟踪层 |

## 2. 核心设计约定

### 2.1 聚合粒度

- **dedupe 维度**：策略（`strategy_id`）
- **依据**：策略与业务是 1:1 关系，因此 `strategy_id` 已隐含业务维度
- **语义**：同一策略下触发的所有告警（满足聚合条件的）关联到同一个 Issue，代表"同一个需要跟进的问题"

### 2.2 策略与活跃 Issue 的数量关系

| 范围 | 关系 |
|------|------|
| 当前设计 | 策略 : 活跃 Issue（待审核 + 未解决）= **1:1** |

"活跃"定义：状态为"待审核"或"未解决"的 Issue。

> 当前需求背景已明确：**仅考虑策略与活跃 Issue 的 1:1 关系**，1:N 关联不在本轮设计与开发范围内。

## 3. 状态机

```
┌─────────────────────────────────────────────────────────────────┐
│                          Issue 状态流转                          │
└─────────────────────────────────────────────────────────────────┘

  [告警触发] ──→  待审核
                  │  负责人=未指派
                  │
          ┌───────┴────────┐
          │                │
     指派负责人        拒绝/标记无效
    （自动流转）       （实例级）
          │                │
          ↓                ↓
        未解决           拒绝/无效
          │                │
     人工标记          下次同策略触发
      已解决           仍创建新 Issue
          │
          ↓
        已解决
          │
    同策略再次触发
          │
          ↓
  新建 Issue（标记"回归问题"，ID 不复用）
```

### 3.1 状态说明

| 状态 | 触发条件 | 负责人 | 说明 |
|------|---------|-------|------|
| **待审核** | Issue 初始创建 | 未指派 | 等待人工介入 |
| **未解决** | 指派负责人后自动流转 | 已指派 | 正在跟进中 |
| **已解决** | 负责人手动标记 | 已指派 | 人工确认问题已处理完毕 |
| **拒绝/无效** | 人工操作，待审核阶段可执行 | — | 实例级，不影响该策略后续 Issue 创建 |

### 3.2 关键规则

- **拒绝为实例级**：拒绝当前 Issue 后，同策略下次触发告警时仍会正常创建新 Issue
- **回归问题**：已解决的 Issue 对应的策略再次触发告警时，新建 Issue 并标记为"回归问题"，ID 不复用
- **历史 Issue 判断依据**：根据"同策略是否曾出现过已解决 Issue"来判断是否为回归问题；在当前 1:1 模型下，同策略即同问题

## 4. Issue 触发与聚合规则（策略配置）

### 4.1 配置入口

在告警策略的"告警处理"面板，新增 **"Issues 聚合"** tab，与"处理套餐"并行，二者可以同时配置（非二选一）。

### 4.2 配置参数

| 参数 | 是否必填 | 默认值 | 说明 |
|------|---------|--------|------|
| **聚合维度** | 必填 | 策略运行时公共维度 | 维度来源复用现有策略语义：单 Item 取 `Item.public_dimensions`，整个策略取 `Strategy.public_dimensions`；配置时只能减少不能新增；用于约束可配置的过滤维度，并在 Issue 创建时固化为配置快照，不参与 Issue 唯一性判断 |
| **条件** | 可选 | 无 | 过滤语义：满足条件的告警才被纳入此 Issue；不满足的告警不影响 Issue 创建；条件方法复用现有策略条件语义（`eq/neq/include/exclude/reg/nreg`），实现入口对齐 access 模块 `ConditionFilter.filter()` + `gen_condition_matcher()` |
| **生效告警级别** | 必填 | 策略已配置的全部级别 | 只有指定级别的告警才触发 Issue 创建/追加 |

### 4.3 聚合逻辑

> 在当前版本中，活跃 Issue 的聚合唯一性只由 `strategy_id` 决定；`aggregate_dimensions` 仅用于过滤语义和配置快照展示。其可配置范围不是策略所有 `agg_dimension` 的并集，而是当前策略 `Strategy.public_dimensions`。

```
告警产生
  │
  ├─ 策略是否配置了 Issues 聚合？ → 否 → 跳过
  │
  ├─ 告警级别是否在生效级别范围内？ → 否 → 跳过
  │
  ├─ 告警是否满足聚合条件？ → 否 → 跳过
  │
  ├─ 当前策略是否有活跃 Issue？
  │     ├─ 有 → 关联到已有 Issue（追加告警事件）
  │     └─ 无 → 创建新 Issue
  │               ├─ 历史是否有已解决的 Issue？ → 有 → 标记"回归问题"
  │               └─ 否则 → 普通新建
  └─ 完成
```

## 5. Issue 字段说明

### 5.1 列表页字段

| 字段 | 说明 |
|------|------|
| **issue 名称** | 创建时由后端按稳定默认规则生成，至少包含告警策略名称；若为回归问题可附带回归标识。后台第一阶段先保证默认值稳定，后续 API / 前端阶段支持人工编辑 |
| **标签** | 继承告警策略的标签 |
| **最早发生时间** | 最早关联告警的产生时间 |
| **最后出现时间** | 最新关联告警的产生时间 |
| **趋势** | 关联告警事件的时间趋势统计图 |
| **影响范围** | 可选快照字段；从关联告警的目标信息汇总得到，主要描述受影响的主机、云区域等基建范围。创建时可先写初值，后续由周期任务重算覆盖；允许为空，不作为流程判断条件 |
| **优先级** | 字段值固定为 P0/P1/P2，中文展示为高/中/低（不与告警级别做映射，默认 P2=低），可手动调整；支持表头筛选 |
| **状态** | 待审核 / 未解决 / 已解决 / 拒绝无效；支持表头筛选 |
| **负责人** | 存储层 `assignee=""` 表示未指派，展示层统一渲染为"未指派"；独立于告警分派；支持表头筛选 |
| **操作** | 标记为已解决 |

### 5.2 批量操作

- 指派负责人
- 标记为已解决
- 修改优先级
- 添加跟进信息

### 5.3 其他功能

- **搜索**：沿用告警中心的筛选器交互
- **导出**：导出 Excel 表格数据

## 6. Issue 详情页

详情页分为 7 个功能区（对应产品截图标注）：

### 区域 1：标题栏

- Issue 标题展示
- 时间范围过滤（影响详情内告警事件的展示范围）
- 全屏操作

### 区域 2：告警事件筛选器

- 沿用告警事件的筛选条件
- 用于过滤详情内展示的关联告警事件

### 区域 3：告警趋势统计

- 可视化展示关联告警事件随时间的分布趋势

### 区域 4：告警事件列表

- 右上角切换：**最新的** / **最早的** / **全部**
- "最新的"和"最早的"对应当前 Issue 实时的代表性告警
- "全部"展示完整的关联告警列表
- 每条告警展示：ID、状态、告警内容、持续时间、维度信息、基础信息（所属业务、首次/告警产生时间、通知人、关注人、告警标签）
- 详情内置图表：主调成功率、维度分析等

### 区域 5：基础信息面板

- 优先级（可修改）
- 负责人（可修改/指派）
- 影响范围
- 最后出现时间 / 最早发生时间
- **标记为已解决**操作按钮

### 区域 6：问题活动

- 人工跟进评论（支持文本输入；评论内容写入活动日志 `content`）
- 变更记录：状态流转、优先级变更、指派负责人等所有 Issue 操作的时间线
- 系统活动以结构化字段表达，不要求在活动日志 `content` 中重复拼接文案

### 区域 7：历史 Issue

- 基于"同策略是否曾出现过已解决 Issue"判断是否曾经发生过该问题
- 展示历史 Issue 列表及其解决时间，辅助判断回归原因

## 7. 外部事项关联（未来方向）

当前版本不包含，规划在后续迭代中支持：

- **关系**：1 Issue : N 外部事项（TAPD Story / GitHub Issue / ITSM 工单）
- **创建方式**：初期为手动关联
- **语义**：一个问题的跟进可能拆解为多个独立的执行事项，例如：
  - 后端修复 bug → TAPD Story
  - 运维配置变更 → ITSM 工单
  - 代码审查 → GitHub PR

## 8. 与现有模块的关系

```
数据源 → access → detect → trigger → alert → converge → fta_action
                                                              │
                                                    ┌─────────┴─────────┐
                                                    │   处理套餐（已有）  │
                                                    │   Issues 聚合（新）│
                                                    └───────────────────┘
                                                              │
                                                        Issue 创建/追加
                                                              │
                                                    人工跟踪 → Issue 关闭
```

Issues 聚合作为 `fta_action` 的新动作类型，与处理套餐并行配置，在告警产生时触发 Issue 的创建或追加逻辑。
