## 简介
本指南面向生产环境的部署与运维，围绕容器化构建、Kubernetes 集群部署、负载均衡与高可用、性能监控与告警、日志与自动化运维展开。内容基于仓库中的 Dockerfile、Helm Chart 文档、部署架构文档与脚本，结合蓝鲸生态与监控平台特性，给出可落地的实施建议与最佳实践。

## 项目结构
- 容器镜像构建：通过多阶段 Dockerfile 构建 Python 运行时与前端静态资源，最终以非 root 用户运行 Django 应用。
- Helm Chart 与 K8s 采集：提供完整的 k8s 指标采集、DataID 路由、Worker 调度与系统内置指标白名单等配置说明。
- 部署角色与组件：明确 web/api/worker 三类角色及其依赖中间件，包括 Celery 双轨制与缓存能力差异。
- 运维脚本：提供发布模式切换与 Django 管理命令入口脚本，便于自动化部署与日常维护。

## 详细组件分析

### 容器化构建流程
- 多阶段构建要点
  - Python 基础阶段：安装常用工具与中文字体，准备运行时环境。
  - Python 构建阶段：使用 uv 管理依赖，隔离构建缓存，加速安装。
  - Node 构建阶段：使用 pnpm 构建前端静态资源，输出 dist。
  - 最终镜像：复制虚拟环境与静态资源，设置非 root 用户与 UTF-8 环境，ENTRYPOINT 使用 tini，CMD 启动 Django。
- 镜像参数与变量
  - 支持通过 ARG 注入基础镜像与 PyPI 源，便于企业内网定制。
  - 版本号写入 VERSION 文件，便于镜像追踪。
- 运行时与安全
  - 非 root 用户运行，降低权限风险。
  - 设置 LC_ALL 与 LANG，确保国际化显示正常。

### Kubernetes 部署与采集配置
- Chart 结构与组件
  - umbrella chart 包含 CRD、内置 exporter、子 chart 与本地 operator/worker。
  - 三类 Worker：DaemonSet（节点本地）、StatefulSet（远端/外部）、Event Worker（k8s 事件）。
- DataID 与路由
  - DataID CRD 定义与 operator 匹配逻辑，支持 scheduledDataID、isSystem 等 annotation。
  - 系统内置指标与自定义指标走不同 DataID，保障数据隔离。
- 指标白名单与 relabelRule
  - kubelet/cAdvisor/probes/resource 等端点的指标重命名与过滤。
  - relabelRule 扩展：v1/workload、v1/node、v2/workload、v3/workload 等，实现维度增强。
- 资源与调度
  - 各组件 CPU/Memory 限制与亲和/容忍配置。
  - StatefulSet Worker HPA 与调度算法（hash/roundrobin）。

### 部署角色与高可用
- 角色划分
  - web：SaaS 前端 API 与 Celery，无 Redis 缓存。
  - api：后台内部 API，继承 worker 的 Redis 配置。
  - worker：alarm_backends 后台任务，独立 Celery App 与队列。
- 中间件依赖
  - MySQL、Redis、Elasticsearch、Kafka、RabbitMQ、InfluxDB、Consul、Transfer、Unify-Query、Grafana。
- Celery 双轨制
  - web Celery 与 alarm_backends Celery 分离，使用不同 Broker 与 App，避免相互干扰。
- 缓存能力
  - 仅 worker/api 角色具备 alarm_backends Redis 缓存配置，web 角色不支持。

## 依赖分析
- 组件耦合
  - web/api 通过 RabbitMQ/Redis 与 alarm_backends 解耦；alarm_backends 使用独立 Celery App 与 Broker。
  - 采集链路通过 Helm Chart 与 operator 解耦，DataID 路由与 relabelRule 保证指标质量与维度。
- 外部依赖
  - MySQL/Redis/ES/Kafka/InfluxDB/Consul 为核心存储与消息通道；Transfer/Unify-Query/Grafana 提供数据处理与可视化。
- 潜在环路
  - 无直接循环依赖；若自定义 DataID 与 isSystem 标注冲突，可能导致匹配不确定性，需规范标注。

## 性能考虑
- 容器镜像与启动
  - 多阶段构建减少镜像体积，缓存 uv 与字体安装提升构建效率。
  - 非 root 用户与 UTF-8 环境减少运行时问题。
- 采集与上报
  - DataID 白名单与 relabelRule 降低指标风暴风险；批量上报参数可调优吞吐。
  - HPA 与调度算法（hash/roundrobin）平衡任务分布。
- 后台任务
  - Celery 双轨制避免 web/api 与 alarm_backends 的资源争抢；队列分离提升稳定性。
- 查询与存储
  - Unify-Query 屏蔽底层存储差异；合理配置 ES/InfluxDB 索引与保留策略，避免热点写入。

## 故障排查指南
- 容器与运行
  - 检查镜像构建日志与 ENTRYPOINT 参数，确认 tini 是否正确初始化。
  - 若页面乱码，检查 LC_ALL/LANG 环境变量是否设置。
- Helm 与采集
  - 确认 DataID CRD 是否存在且标注正确（isSystem/scheduledDataID）。
  - 查看 operator 日志，定位匹配失败或 ChildConfig 下发异常。
  - 检查 Worker Pod 的 reloader 是否成功写入 child_configs 并触发热重载。
- 部署角色与缓存
  - web 角色无法使用 alarm_backends 缓存，避免在 web 进程中调用相关缓存工具。
  - api/worker 角色需确保 Redis 配置完整，DB 选择正确。
- 发布与运维
  - 使用 release.sh 切换模式并 reload，观察 supervisor 状态。
  - 使用 manage.sh 执行迁移、收集静态资源等命令，确保环境变量正确。
