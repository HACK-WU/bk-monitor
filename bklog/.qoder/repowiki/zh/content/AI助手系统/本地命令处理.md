## 项目结构
本地命令处理系统位于应用模块中，通过装饰器注册命令处理器，并在应用启动时自动完成注册。命令处理器基于统一的基类，遵循相同的生命周期：参数提取、内容处理、模板渲染与结果返回。

## 架构总览
本地命令处理系统采用“装饰器注册 + 统一基类 + 模板渲染”的架构模式。命令通过装饰器注册到全局注册表，运行时按需调用对应处理器进行内容处理与结果渲染。

```mermaid
sequenceDiagram
participant Caller as "调用方"
participant Registry as "命令注册表"
participant Handler as "具体命令处理器"
participant Search as "日志查询子系统"
Caller->>Registry : 请求执行命令(命令名, 参数)
Registry->>Handler : 定位处理器并实例化
Handler->>Handler : 提取上下文变量
alt 需要查询上下文
Handler->>Search : 构建查询参数并发起查询
Search-->>Handler : 返回上下文日志列表
end
Handler->>Handler : 处理内容(清洗/拼接/模板渲染)
Handler-->>Caller : 返回处理结果
```

### 命令注册机制
- 应用启动时导入命令处理器模块，触发装饰器注册。
- 装饰器将命令名与处理器类绑定，建立命令名到处理器类的映射。
- 注册完成后，系统可通过命令名定位并调用对应处理器。

```mermaid
flowchart TD
Start(["应用启动"]) --> Import["导入本地命令处理器模块"]
Import --> Decorator["@local_command_handler 装饰器执行"]
Decorator --> Register["注册到全局注册表"]
Register --> Ready["等待命令调用"]
```

### 执行流程与结果反馈
- 参数提取：从上下文消息中抽取命令所需的变量。
- 内容处理：根据命令类型执行相应逻辑（如查询上下文、拼接模板等）。
- 模板渲染：使用 Jinja2 模板引擎渲染最终结果字符串。
- 结果返回：将渲染后的文本返回给调用方。

## 故障排除指南
- 常见问题
  - 命令未生效：确认应用初始化是否导入了命令处理器模块。
  - 查询失败：检查索引集是否存在、字段快照是否可用、业务 ID 映射是否正确。
  - 输出为空：确认上下文查询返回结果非空，或检查字段清洗规则是否过于严格。
- 排查步骤
  - 查看日志记录器输出，定位异常发生点。
  - 校验输入参数类型与完整性。
  - 逐步缩小问题范围，先验证查询链路，再检查模板渲染。
